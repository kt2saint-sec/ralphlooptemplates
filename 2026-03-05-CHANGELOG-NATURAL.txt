WHAT WAS FOUND AND FIXED

PROMISE DETECTION OVERHAUL. The original ralph-loop plugin detects completion
promises by looking for XML promise tags in Claude's output using a Perl regex.
The problem is that Claude Code's rendering pipeline STRIPS XML and HTML tags
from transcript files before they are written to disk. This means the promise
tags never appear in the transcript, and the loop can never detect completion.
Our fix replaces the XML tag approach entirely with plain text exact line
matching using grep -Fx. We also added an auto-generated passphrase system
using the format WORD NNNN WORD NNNN WORD NNNN, where words are drawn from
three semantic domains (materials like GRANITE, animals like OSPREY, and science
terms like COSINE). This produces approximately 8 trillion combinations, making
false positive detection effectively impossible. User-provided promises are
prefixed with the passphrase using a double colon separator.

SESSION ID FIX. The original plugin uses CLAUDE_SESSION_ID as the primary
session identifier, falling back to PPID (parent process ID). We discovered
that CLAUDE_SESSION_ID DOES NOT EXIST as an environment variable. PPID is
also unreliable because the setup script and stop hook run as separate
processes with different parent PIDs. Our fix extracts session_id from the
hook input JSON that Claude Code pipes via stdin to stop hooks. This field
is documented in the official Claude Code hooks documentation. The setup
script generates a temporary ID using uuidgen, and the stop hook renames the
state file to the real hook session_id on the first iteration.

STATE FILE LOOKUP PERFORMANCE. The original plugin does a direct file lookup
using PPID, which breaks because PPID is wrong. Our version does O(1) direct
lookup by hook session_id, with a glob plus ls -t fallback for the first
iteration (before the rename happens). This also supports multi-terminal
scenarios where multiple ralph loops run simultaneously.

MULTI TERMINAL CONTAMINATION. The original plugin creates a single state file
named ralph-loop.SESSION_ID.local.md using PPID, which means different
terminals get different PPIDs and cannot find each other's state files. But
because PPID is unreliable, the wrong terminal might pick up the wrong state.
Our fix uses the hook-provided session_id which is consistent and unique per
session, plus a dual glob pattern that supports both the original format
(ralph-loop.local.md) and our session-scoped format (ralph-loop.SESSION.local.md).

OUTPUT EXTRACTION IMPROVEMENT. The original plugin parses the transcript JSONL
file to find the last assistant message. Our version first checks the
last_assistant_message field from the hook input JSON, which Claude Code
provides directly. This avoids JSONL parsing entirely. If the field is empty
or missing, we fall back to transcript parsing for backward compatibility.

FRONTMATTER AWK BUG. The original plugin uses an awk pattern that skips ALL
lines matching the triple dash separator. This corrupts prompts that contain
markdown horizontal rules or YAML-like separators in their content, because
those lines get eaten by the awk filter. Our fix changes the awk to only skip
the first two triple dash lines (which delimit the YAML frontmatter), preserving
all subsequent triple dash lines as prompt content.

NULLGLOB BYPASS FIX. In bash, the nullglob shell option only applies to glob
PATTERNS containing wildcard characters like asterisk, question mark, or
bracket expressions. A literal path like ralph-loop.local.md bypasses nullglob
entirely and always appears in array expansions even if the file does not exist.
Our fix uses a character class trick: ralph-loop.loca[l].md, which is
functionally identical but counts as a glob pattern, making nullglob work
correctly.

PLUGIN CACHE OVERWRITE PROTECTION. When Claude Code updates a plugin from the
marketplace, it creates a new cache directory and marks the old one with an
.orphaned_at file. This silently overwrites any local customizations. We added
two tools: cache-watchdog.sh (a SessionStart hook that compares repo files
against cached copies and warns on mismatch) and cache-sync.sh (copies patched
files from the repo to the active cache directory, handling the scripts to hooks
path mapping automatically).

LEARNINGS DEFAULT CHANGED. The default for learnings_enabled was changed from
true to false to reduce noise in short loops where per-iteration retrospectives
add overhead without benefit.

CONSOLIDATION PROMPT FIX. The consolidation prompt that fires when a loop
completes was updated to use plain text completion signals instead of XML
promise tags, matching the overall change in promise detection strategy.

FLOCK PROTECTED RENAME. The state file rename on first iteration (from uuidgen
ID to hook session_id) is protected by flock to prevent race conditions when
multiple terminals hit the glob fallback simultaneously.

CANCEL COMMAND SESSION SCOPING. The cancel-ralph command was updated to be
session-scoped, reading the session ID from the state file frontmatter to
delete only the current session's files rather than all sessions.


WHAT REMAINS UNVERIFIED OR UNKNOWN

HOOK JSON FIELDS IN LIVE FIRE. The session_id and last_assistant_message fields
from the hook input JSON have been verified to work via direct pipe invocation
(echo JSON | bash stop-hook.sh), which is functionally equivalent to how Claude
Code invokes hooks. However, they have NOT been observed in an actual hook fire
event. Debug dump lines exist in the stop hook (commented out at lines 27-28)
that can be uncommented to capture the raw JSON to /tmp/ralph-hook-debug.json
for verification. This requires syncing to cache, starting a NEW session,
running a ralph loop for 2 iterations, then inspecting the dump file.

STOP_HOOK_ACTIVE FIELD BEHAVIOR. The stop_hook_active field is extracted from
the hook JSON but intentionally NOT used as an exit guard. On iteration 2 and
beyond, this field would always be true because the loop works by repeatedly
blocking stop events. Using it as an exit guard would kill the loop after the
first iteration. This decision is documented but the actual behavior of the
field has not been observed in a live hook fire.

LOCAL PLUGIN INSTALLATION VIABILITY. It is unknown whether changing the
installPath in installed_plugins.json to point at a local repo directory
(instead of the cache) would be respected by Claude Code. The cache system
may validate paths, check for .orphaned_at markers, or require specific
directory structures. This needs testing.

XML TAG SURVIVAL IN TRANSCRIPTS. It is unconfirmed whether XML promise tags
like <promise>TEXT</promise> survive in the transcript JSONL file or are
stripped by Claude Code's rendering pipeline before writing. If they survive,
the original plugin's Perl regex approach would work for transcript parsing
but still fail for last_assistant_message (which contains rendered text). If
they are stripped, the original approach fails everywhere.

CACHE SYNC MID SESSION LIMITATION. Syncing files to the plugin cache mid
session does NOT affect running hooks. Claude Code loads hooks at session start
and does not re-read them. This is documented behavior but means any fix
applied via cache-sync requires starting a new Claude Code session to take
effect. There is no workaround for this.

PASSPHRASE GENERATION FIX (2026-03-06, session 15)

The WORD NNNN WORD NNNN WORD NNNN passphrase system had a fundamental flaw:
LLMs have consistent token probability biases. When told to pick a random word
from a list, they gravitate toward the same high-salience tokens every time.
MARBLE, CONDOR, and LATTICE appeared disproportionately across sessions. This
is not a prompting problem — it is a property of next-token prediction. No
amount of instruction tuning fixes it.

The fix replaces the entire word-array system with a Bash tool call to
/dev/urandom: echo "RALPH-$(head -c 24 /dev/urandom | xxd -p | tr -d '\n')"
This produces a RALPH- prefix followed by 48 hex characters of TRUE OS
randomness. Zero LLM bias. The RALPH- prefix prevents false positive matches
against hex strings that appear naturally in code output. Compatible with the
existing grep -Fx detection in stop-hook.sh.

PROMPT GENERATION RELIABILITY FIX (2026-03-06, session 15)

The /ralphtemplatetest and /ralphtemplate commands sometimes failed to output
the prompt when the user's arguments looked like a question or diagnostic
request rather than an implementation task. Two fixes applied:

1. Added a CRITICAL guard instruction: the prompt MUST always be generated
regardless of argument content. The user's entire argument text goes into the
TASK field verbatim. Zero exceptions.

2. Replaced the --- template delimiters (identical to YAML frontmatter
delimiters) with unambiguous === TEMPLATE START === and === TEMPLATE END ===
markers. This eliminates parsing confusion between frontmatter boundaries and
template content boundaries.

Added docs/screenshot-*.png to .gitignore to prevent binary bloat.


DECISIONS MADE

55 architectural decisions across 15 development sessions are documented in
LEARNINGS.md. Key decisions include using grep -Fx over Perl regex, using
uuidgen over PPID, implementing flock for atomic renames, choosing to NOT use
stop_hook_active as an exit guard, replacing the word-array passphrase system
with /dev/urandom hex hashes, and migrating from plugin to hybrid architecture.
