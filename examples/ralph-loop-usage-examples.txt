Ralph Loop Usage Examples
=========================

These are sanitized examples showing how to invoke /ralph-loop with the
Boris Antagonist Method rules embedded in the prompt.

Example 1: Full-Stack Application Build
----------------------------------------

/ralph-loop:ralph-loop "The goal is to create [your-application].
You are an orchestrator that is not to code. You are to delegate 6
parallel agents/subagents allowing offloading context from this session.

You are to interactively test the auth system with [your-test-method]
to ensure it is fully functional. Create tests as you develop, expecting
them to fail based on the stage you are in, and as you develop, ensure
to test. Update the CLAUDE.md as you develop to avoid making the
mistakes twice.

When you test the application, whatever is non-functional is something
to have a subagent document to create a report. Ensure that as the
orchestrator you assign the best agents for whichever task you are on,
and work through the stages.

Once you are developing the frontend, use the chrome MCP so the user
can see the dev progress in the browser.

## Potential Risks Not Verified:
- [Component]: [Why it might break] -> [Plan to address]
- [Integration]: [Assumption made] -> [Verification needed]"
  --max-iterations 30 --completion-promise "DONE"


Example 2: Minimal Loop (Bug Fix)
----------------------------------

/ralph-loop "Fix the [describe-bug] in [your-module]. Run the test suite
after each fix attempt." --max-iterations 10 --completion-promise "DONE"


Example 3: Refactor with Proof
-------------------------------

/ralph-loop "Refactor [your-module] to use [new-pattern]. Keep all
existing tests passing. Add new tests for the refactored code.
Prove each change works before moving to the next."
  --max-iterations 20 --completion-promise "ALL TESTS PASSING"


Example 4: With Boris Rules Embedded
--------------------------------------

/ralph-loop:ralph-loop "Before any PR or merge request:
- Interrogate the developer on design choices:
  - Why did you choose this approach over [alternative]?
  - What edge cases does this handle? What doesn't it handle?
  - How does this affect performance/memory/security?
  - What happens if [failure scenario]?
- Block the PR until satisfactory answers are provided

[your-actual-task-description-here]

Format: ## Potential Risks Not Verified:
- [Component]: [Why it might break] -> [Plan to address]
- [Integration]: [Assumption made] -> [Verification needed]"
  --max-iterations 40 --completion-promise "DONE"


Key Patterns
=============

1. ALWAYS include --max-iterations to prevent infinite loops
2. ALWAYS include --completion-promise so the loop knows when to stop
3. Embed the Boris rules in the prompt itself for self-enforcing quality
4. Use the orchestrator pattern (delegate, don't code) for complex tasks
5. Include the "Potential Risks" format so each iteration documents unknowns
