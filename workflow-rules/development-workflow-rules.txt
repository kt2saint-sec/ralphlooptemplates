DEVELOPMENT WORKFLOW RULES - CLAUDE CODE AGENT PROTOCOL

These rules enforce rigorous development practices for Claude Code agents.
Add them to your CLAUDE.md or project instructions.

## Development Workflow Rules

### 1. Code Review Gate (No PR Without Passing Interrogation)

Before any PR or merge request:
- **Interrogate the developer** on design choices:
  - "Why did you choose this approach over [alternative]?"
  - "What edge cases does this handle? What doesn't it handle?"
  - "How does this affect performance/memory/security?"
  - "What happens if [failure scenario]?"
- **Block the PR** until satisfactory answers are provided
- Document the Q&A in the PR description

### 2. Proof of Functionality Required

Before any commit or push:
- **Demonstrate working code** with actual output
- **Show diff behavior** between current state and changes:
  git diff main..HEAD --stat
  git diff main..HEAD -- [specific files]
- **Provide evidence**: test results, screenshots, logs, or terminal output
- If tests don't exist, **create them first** before claiming it works

### 3. Elegant Solution Protocol

After completing a first-pass implementation:
- **Pause and evaluate**: "Is this the elegant solution or a working hack?"
- If mediocre: **Scrap and rebuild** using accumulated context
- Before rebuilding, document:
  - What was learned from the first attempt
  - What the elegant solution looks like
  - Why it's better (performance, readability, maintainability)

### 4. Unknown Impact Disclosure

At the end of every implementation loop, **explicitly report**:
- Files touched but not fully tested
- Dependencies that may have changed behavior
- Areas that *might* break but weren't verified
- External integrations not tested in this cycle
- Assumptions made without verification

Format:
  ## Potential Risks Not Verified:
  - [ ] [Component]: [Why it might break] -> [Plan to address]
  - [ ] [Integration]: [Assumption made] -> [Verification needed]

**Address these in the next loop before new features.**

### 5. No Push/Commit Without Proof

**Hard rule**: Nothing gets committed or pushed without:
1. Passing tests (or explanation why tests can't run)
2. Manual verification output shown
3. Diff reviewed and explained
4. Unknown impacts disclosed
5. Developer passed code review interrogation

Workflow:
  # Before ANY commit:
  git status                    # Show what's staged
  git diff --cached            # Show exact changes
  # [Run tests and show output]
  # [Show manual verification]
  # [List potential risks]
  # THEN commit

## Agent-Specific Rules

### Before Any Agent Commits:
1. Run: git diff main..HEAD
2. Run relevant tests: python -m pytest tests/ -v (or your test runner)
3. Show output proving functionality
4. List files changed but not tested
5. Get approval from orchestrator agent

### Inter-Agent Handoffs:
- Receiving agent must **verify** previous agent's work
- No assumptions - re-run proofs
- Document any discrepancies found

### Bug-Fix Specific:
- Show the bug (reproduction)
- Show the fix (diff)
- Show it's fixed (test passing)
- Show nothing else broke (regression tests)

## Quick Reference Commands

Use these phrases with your Claude Code agents:
- "Grill me on this before PR" -> Interrogation mode (agent asks hard questions)
- "Prove this works" -> Must show diffs, test output, evidence
- "Elegant solution - start over" -> Full rebuild with context retained
- "What might break that you didn't test?" -> Unknown impact disclosure
- "Commit gate check" -> Full verification before any git operation
