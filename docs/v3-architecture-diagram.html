CORE Execution Phases (from ralphtemplatetest.md) 4 phases + sandbox post-completion verification
PHASE 1 Challenger & Proxy Negotiate Loops until Challenger satisfied or objections exhausted
Challenger raises 5+ ambiguities. Proxy answers from codebase/docs. If Proxy <75% certain, Researcher investigates first.
PHASE 1.5 Tester Creates Tests in Sandbox Only with /ralphtemplatetest (skipped if TESTINGOFF)
Tester reads Challenger's analysis + Proxy's decisions. Writes tests in /mnt/nvme-fast/claude-workspace/sandbox/ralph-test-sandbox-SESSION_ID/. Tests verify BEHAVIOR, designed to fail initially. Builder has NOT seen this code. Challenger reviews edge case coverage.
PHASE 2 Builder Implements (Ralph Loop Iteration) Stop hook re-feeds prompt each iteration
Builder implements blind to test code. Challenger reviews after milestones. Researcher consulted when <75% certain. Each iteration: status, root cause, research, tests, fix, result, Challenger notes.
PHASE 3 Builder Runs Sandbox Tests
Builder runs Tester's suite from sandbox. Tests fail = fix IMPLEMENTATION (not tests). Challenger checks if failures indicate design problem. Also runs existing project tests. Reports sandbox results (pass count / total).
PHASE 4 + POST-COMPLETION Verify, Output Passphrase, Final Sandbox Run
Builder answers completion checklist honestly. Outputs passphrase on its own line. Stop hook detects via grep -Fx. Final sandbox test run. PASS = promise honored, sandbox rm -rf. FAIL = promise REVOKED, Builder loops back to Phase 2.

STOP HOOK Session-Aware Loop Controller
The stop hook is the engine that makes Ralph Loop work. It fires on EVERY Claude Code session exit (no matcher support for Stop hooks). A fast-exit guard using nullglob checks for state files BEFORE reading stdin, so non-Ralph sessions have near-zero overhead.
State file frontmatter (YAML):
active: true | iteration: 3 | max_iterations: 15
completion_promise: "RALPH-a3f8c9..." | learnings_enabled: true
session_id: "abc123" | started_at: "2026-03-07T..."

Hook output JSON (blocks exit):
{ "decision": "block", "reason": "<prompt>", "systemMessage": "..." }
Session ID resolution (stop-hook.sh:42-95):
1. Primary: O(1) direct lookup by .claude/ralph-loop.HOOK_SESSION_ID.local.md
2. Fallback: nullglob + ls -t (first iteration when setup used uuidgen)
3. On fallback: flock-protected rename to hook session_id + frontmatter sed update
4. Learnings file renamed to match: .claude/ralph-learnings.HOOK_SESSION_ID.md

PERF I/O Pressure Optimization (Session 16) reduce-io-pressure.sh — separate sandbox writes from root NVMe
Problem: Root NVMe Contention
Root NVMe handling:
- journald persistent logs (unbounded)
- /tmp sandbox writes
- Docker container I/O
- System + app writes
All on a single NVMe drive
All competing for same write queue
Fix: 3 Optimizations
1. journald: capped 2GB persistent
   (NOT volatile — crash logs survive reboot)
2. Workspace: /mnt/nvme-fast/
   claude-workspace/{sandbox,tmp,builds}
3. /tmp: tmpfs 16GB in fstab
   (fstab-only, requires reboot to activate)
Result: Split Write Queues
Root NVMe: system + apps only
Fast NVMe: Docker, workspace, sandbox
tmpfs: /tmp in RAM (post-reboot)
