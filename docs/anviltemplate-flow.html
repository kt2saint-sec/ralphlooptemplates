/anviltemplate

Takes a task description. Outputs a self-contained adversarial prompt with 6 compartmentalized roles.

What It Does

You give it a short task. It generates a long, structured prompt that forces Claude to operate as 6 isolated roles with strict information barriers — preventing every common failure mode of AI-assisted coding.

You type
/anviltemplate Build a JWT auth
system with refresh tokens
→
It generates (~250 lines plain text)
You are an orchestrator that creates
and delegates the best agents and
subagents for the task.

TASK: Build a JWT auth system with
refresh tokens

ORCHESTRATION RULES
You will operate six internal roles...
[ EVALUATOR • CHALLENGER • PROXY
  RESEARCHER • TESTER • BUILDER
  + phases + passphrase + checklist ]
The generated prompt is piped into /anvil-loop which feeds it back to Claude on every iteration, creating a self-correcting development cycle.

Generator Pipeline

When you run /anviltemplate, these steps execute in sequence:

1
Parse arguments
User's text becomes TASK_DESCRIPTION verbatim. Detects TESTINGOFF flag to toggle Tester role on/off.
2
Generate passphrase
Calls /dev/urandom via Bash — true OS randomness, zero LLM token bias.
ANVIL-$(printf '%08x' "$(date +%s)")-$(head -c 20 /dev/urandom | xxd -p)
3
Assemble prompt
Substitutes TASK_DESCRIPTION and GENERATED_PASSPHRASE into the template. All 6 roles, phases, iteration format, completion checklist, and assumptions are included as plain text (no markdown).
4
Output to user
Displays the full prompt for copy-paste, plus the passphrase separately, plus the suggested /anvil-loop command.
5
DOCUMENTOR saves files
Writes raw prompt to anvil-prompt-YYYY-MM-DD-HHMM.txt and spawns a haiku agent for the summary file with metadata.

Modular: TESTINGOFF Toggle

The Tester role is modular. Add TESTINGOFF to your arguments to generate a 5-role prompt without clean-room testing.

6-Role (Default)
Evaluator • Challenger • Proxy • Researcher • Tester • Builder
Full adversarial + clean-room testing + sandbox isolation
/anviltemplate Build auth system
5-Role (TESTINGOFF)
Evaluator • Challenger • Proxy • Researcher • Builder
Adversarial challenge without sandbox testing overhead
/anviltemplate Build auth system TESTINGOFF
When to use TESTINGOFF: Quick fixes, config changes, documentation tasks, or when the project already has comprehensive tests. The Challenger still stress-tests the approach — you just skip the isolated sandbox tests.

Anatomy of the Generated Prompt

Every generated prompt contains these blocks in order. Each block constrains a specific role with explicit boundaries on what it knows, does, and cannot do.

TASK
Your task description, inserted verbatim. This is the only input the system needs.
ROLE 0: EVALUATOR
Scores complexity across 5 dimensions (specs, scope, risk, deps, ambiguity). Assigns tier: LIGHT → MAXIMAL. Exits immediately — no lingering authority. Tier governs all other roles' intensity.
ROLE 1: BUILDER
Primary implementer. Works within EVALUATOR's iteration budget. Below 75% certainty → delegates to Researcher. Cannot write code until Tester creates tests. Sees test results, never test source code.
ROLE 2: CHALLENGER
Boris-method antagonist. Raises tier-scaled objections (3 to 12+). Routes questions to Proxy, never to user. Reviews after every milestone. THOROUGH+ reviews each spec separately. Cannot approve — only pressure.
ROLE 3: PROXY
Human stand-in. Answers by reading codebase, CLAUDE.md, package.json. Below 75% certainty → delegates to Researcher. Flags low-confidence answers as REVIEWABLE. Never says "ask the user."
ROLE 4: RESEARCHER
Independent fact-finder. Uses subagents, web search, MCP servers (context7, brave, fetch, github). No stake in outcome. Reports findings with sources and confidence. Cannot make decisions.
ROLE 5: TESTER
Clean-room test designer. Enters with goals document only — no discussion context, no rejected alternatives. Writes behavior tests in isolated sandbox BEFORE implementation. Omitted when TESTINGOFF.
EXECUTION FLOW
Phase 0 (assess) → Phase 1 (challenge) → Phase 1.5 (test design) → Phase 2 (implement) → Phase 3 (test & iterate) → Phase 4 (oral defense). Per-spec cycling for THOROUGH+.
COMPLETION
Passphrase (ANVIL-hex) — output on its own line only when genuinely done. Revoked if final test run fails. Iteration format standardized for each loop cycle.
CHECKLIST
8 questions the Builder must answer honestly before declaring done. 4 assumptions the user is making that must be confirmed or corrected.

EVALUATOR
Reads task → scores 5 dimensions → assigns tier → exits permanently
Delegates to: nobody. One-shot.
CHALLENGER
Stress-tests approach. Edge cases, security, scale, naming, deps. Blocks if unsatisfied.
Questions → Proxy (never user)
PROXY
Reads codebase + docs. Makes judgment calls. Flags <70% answers as REVIEWABLE.
<75% certainty → Researcher

Subagents, web search, MCP servers, source code. Reports facts with citations.
Delegates to: nobody. Reports back.

Enters clean room with goals only. Writes behavior tests in sandbox before code exists.
Communicates with: nobody. Isolated.
BUILDER
Implements against goal. Gets graded test results (not source). Iterates until 100% pass.
<75% certainty → Researcher

Output Files

anvil-prompt-2026-03-11-0300.txt
Raw prompt text. Pipe directly into anvil-loop:
/anvil-loop "$(cat anvil-prompt-*.txt)"
anvil-prompt-2026-03-11-0300-summary.txt
Metadata: task excerpt, passphrase, suggested command, date. Written by haiku agent.
Terminal output
Full prompt for copy-paste, passphrase shown separately, suggested /anvil-loop command.

Usage

Full 6-role (default)
/anviltemplate Build a REST API for
user management with JWT auth

# then pipe into loop:
/anvil-loop "$(cat anvil-prompt-*.txt)" \
  --max-iterations 20 \
  --completion-promise "PASSPHRASE"
Without testing
/anviltemplate Fix the token refresh
bug in auth.ts TESTINGOFF

# same pipe, fewer roles:
/anvil-loop "$(cat anvil-prompt-*.txt)" \
  --max-iterations 10 \
  --completion-promise "PASSPHRASE"
