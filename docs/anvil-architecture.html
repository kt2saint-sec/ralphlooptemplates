2. The 6-Role Template

Information Compartmentalization

No single role holds the full picture. Each knows only what it needs. Bias cannot propagate across boundaries.

EVALUATOR
Knows: Task complexity — nothing else
Does: Scores difficulty across 5 dimensions, sets tier, exits
Cannot: Implement, challenge, or decide
Scores & exits
CHALLENGER
Knows: The goal + Builder's proposed approach
Does: Adversarial stress-testing, gates the Builder
Cannot: Approve, decide, or implement
Gate only
PROXY
Knows: Goal + codebase conventions + project docs
Does: Answers as human stand-in, makes judgment calls
Cannot: See tests, write code, override Challenger
Human stand-in
RESEARCHER
Knows: Only the specific question asked
Does: Finds facts from docs, web, MCP — reports with citations
Cannot: Make decisions, has no stake in outcome
Unbiased
TESTER
Knows: Goals document only — no discussion, no rejected alternatives
Does: Writes behavior tests alone in sandbox BEFORE implementation
Cannot: See Builder's code, communicate with other roles
Clean room
BUILDER
Knows: Goal + approach (survived Challenger) + test results
Does: Implements in isolation, iterates against test feedback
Cannot: See test source code, renegotiate goal
Implements

3. Adaptive Complexity

The Evaluator reads the task and scores across 5 dimensions: spec count, scope, risk level, dependency depth, and ambiguity. This score determines how hard every other role pushes.

Tier Challenger Tester Iteration Budget When
LIGHT 3+ objections 3-5 tests 5 Single spec, single file, low risk
STANDARD 5+ objections 5-10 tests 10 2-3 specs, single module
THOROUGH 7+ per-spec, proof 10-15 tests 15 3-5 specs, multi-module
RIGOROUS 10+ blocking, proof 15-20 tests 20 5+ specs, full-stack
MAXIMAL 12+ with proof 20+ security tests 30 Cross-system, production, security

For THOROUGH and above, the Challenger reviews each spec separately, the challenge-test-build cycle repeats per spec, and the Challenger demands proof for every claim — assertions alone are rejected. The tier can only escalate — never downgrade.

4. The Passphrase

Every session generates a unique completion signal from OS randomness. No LLM token bias. Exact full-line match detection.

Example passphrase:
ANVIL-00000000-a3f7b2c94e1d08f6a2b3c5d7e1f4a09b2c4d6e8f
Prefix
ANVIL-
Epoch Hex
00000000
Separator
-
/dev/urandom (40 hex chars)
a3f7b2c94e1d08f6a2b3c5d7e1f4a09b2c4d6e8f
Generation
echo "ANVIL-$(printf '%08x' "$(date +%s)")-$(head -c 20 /dev/urandom | xxd -p | tr -d '\n')"
Detection
grep -Fx "$COMPLETION_PROMISE"
Exact full-line match — no regex, no partials
Why not words? LLMs have token bias — word-based passphrases produced repeats (MARBLE, CONDOR, LATTICE). OS randomness via /dev/urandom has zero bias. The epoch hex is decodable for audit: printf '%d\n' 0x00000000
